# Методика исследования DriftGuard

## Цель

Оценить, может ли контекстно-зависимый выбор профиля чувствительности повысить качество обнаружения drift-а в потоках технических метрик по сравнению с фиксированными профилями.

Исследуемая адаптивная стратегия анализирует только начальный baseline-фрагмент потока. Она не получает тип сценария, момент начала drift-а или ожидаемые интервалы. По коэффициенту вариации и лаговой автокорреляции выбирается один из профилей:

- `AGGRESSIVE` для стабильного baseline с малой относительной дисперсией;
- `CONSERVATIVE` для шумного или выраженно автокоррелированного baseline;
- `BALANCED` для промежуточных случаев.

## Гипотезы

**H0:** средний F1 адаптивной стратегии не превосходит средний F1 фиксированных профилей при сопоставимой задержке обнаружения.

**H1:** адаптивная стратегия повышает средний F1 или снижает количество false positives при сопоставимой задержке обнаружения хотя бы для части классов потоков.

Это проверяемая инженерно-научная гипотеза. Наличие реализации само по себе не доказывает научную новизну; окончательное утверждение требует сравнения с опубликованными методами adaptive drift detection.

## План эксперимента

Для каждой комбинации выполняются парные испытания всех стратегий:

- тип потока: step drift, pulse spike, throughput drop, gradual drift, seasonal no-drift;
- множитель шума;
- множитель величины эффекта;
- random seed;
- стратегия: `AGGRESSIVE`, `BALANCED`, `CONSERVATIVE`, `ADAPTIVE`.

Парный дизайн означает, что стратегии получают один и тот же сгенерированный поток для каждой комбинации параметров. Это снижает влияние случайной вариации и позволяет сравнивать методы в одинаковых условиях.

## Метрики

- `precision = TP / (TP + FP)`;
- `recall = detected intervals / expected intervals`;
- `F1 = 2 * precision * recall / (precision + recall)`;
- false-positive events на 1 000 наблюдений;
- первая задержка обнаружения в samples;
- detection rate;
- распределение профилей, выбранных adaptive strategy.

Для среднего F1 выводится 95% confidence interval:

```text
mean(F1) +/- 1.96 * standardError(F1)
```

Используется normal approximation. Для итоговой ВКР рекомендуется не менее 30 seeds либо bootstrap confidence intervals.

## Воспроизводимость

В отчёте сохраняются:

- базовый seed;
- количество repetitions;
- число samples;
- список сценариев;
- множители шума и эффекта;
- результаты каждого trial;
- агрегаты и confidence intervals.

CSV предназначен для внешнего анализа в Python/R, Markdown — для приложений к ВКР и review notes.

## Интерпретация

Нельзя выбирать победителя только по F1. Практически значимый результат должен учитывать одновременно:

- F1 и recall;
- false positives;
- detection delay;
- устойчивость по seeds;
- особенности no-drift seasonal scenario.

Предпочтительно строить Pareto-анализ `F1 / delay / FP`, а не сводить все свойства к одному произвольному score.

## Угрозы валидности

1. Сценарии синтетические и не полностью моделируют production telemetry.
2. Текущая runtime-конфигурация сравнивает профили Page-Hinkley, а не все алгоритмы библиотеки.
3. Правила adaptive selector заданы эвристически и должны быть подтверждены независимой выборкой.
4. Normal-approximation CI неточен при малом числе repetitions.
5. Параметры сценариев и selector нельзя подбирать по одной и той же итоговой выборке.

Для усиления ВКР следует добавить реальные или публичные telemetry datasets, hold-out набор сценариев и статистическое сравнение Friedman/Wilcoxon с коррекцией Holm.
